# StellarNet Lite

> 一个面向 Unity 中小型多人联机项目的轻量级房间网络框架  
> 核心理念：**服务端权威、协议事件驱动、房间组件化、回放沙盒化**

---

## 项目简介

**StellarNet Lite** 是一套基于 Unity 的轻量级联网框架，适用于 **房间制、对局制、中小型商业项目** 的快速开发与长期维护。

它并不追求“黑盒自动同步”，而是强调：

- **数据流转清晰可追踪**
- **服务端绝对权威**
- **客户端表现与业务彻底解耦**
- **按房间组件横向扩展玩法**
- **支持回放、重连、房间 GC 等生产级基础能力**

如果你正在开发这类项目：

- 房间对战
- 休闲联机
- 棋牌桌游
- 小型竞技玩法
- 带录像与断线重连的房间游戏

那么 StellarNet Lite 会比“到处 SyncVar”“逻辑全堆一个房间类里”的方案更适合长期迭代。

## 核心特性

### 1. 服务端绝对权威
客户端只负责：

- 采集输入
- 发送请求
- 接收权威同步
- 刷新表现

所有核心业务状态都由服务端决定，例如：

- 房间状态
- 成员状态
- 血量
- 行为结果
- 结算结果

---

### 2. 显式协议事件驱动
拒绝黑盒自动同步，所有状态变化都通过清晰协议流转：

客户端请求 -> 服务端校验 -> 服务端修改状态 -> 服务端广播 -> 客户端刷新表现

这意味着：

- 消息来源清楚
- 状态流向清楚
- 逻辑更容易排查
- 更适合团队协作

---

### 3. 房间组件化扩展
每个房间玩法都可以拆成独立 `Room Component`：

- 房间设置组件
- 战斗组件
- 表情组件
- 聊天组件
- 投票组件
- 语音组件

建房时通过 `ComponentIds` 动态装配。  
新增玩法时，不需要把所有逻辑塞进一个巨型房间类里。

---

### 4. 客户端 Service / View 解耦
客户端房间组件只负责：

- 接收网络同步
- 做基础校验
- 转发事件
- 维护轻量缓存

View 层只负责：

- UI 刷新
- 特效播放
- 插值移动
- 输入采集

这样可以显著降低 UI 与网络层耦合。

---

### 5. 回放沙盒模式
内置 `ReplayRoom` 概念。  
回放不是“在线看房间”，而是客户端本地重建出来的独立沙盒：

- 支持录像下载与播放
- 支持暂停、倍速、Seek
- 自动隔离真实在线房间消息
- 回放状态下禁止真实发包

---

### 6. 断线重连与状态恢复
框架支持：

- 登录态恢复
- 房间重连确认
- 客户端先装配组件再接收快照
- 服务端按房间上下文精确恢复

重点是：  
**重连流程不依赖黑盒状态复制，而是走清晰的握手与快照恢复机制。**

---

### 7. 底层 Seq 防重放
客户端发包自动附带递增序列号。  
服务端按 Session 执行防重放拦截，避免：

- 重复点击
- 重复请求
- 旧包污染状态机

---

### 8. 生产级治理能力
除了联机基础功能外，框架还包含：

- 协议 ID 冲突扫描
- 房间生命周期 GC
- 空房间熔断清理
- 离线 Session GC
- 录像文件滚动清理
- 编辑器配置窗口

## 设计目标

StellarNet Lite 适合以下目标场景：

- **100~200 CCU** 量级的中小型多人项目
- 需要 **可维护性高于短期开发爽感**
- 团队规模在 **3~8 人** 左右
- 希望业务模块可以 **横向独立扩展**
- 不希望项目后期演化成“网络层和表现层一坨”的状态

它不是为了替代所有大型 MMO 网络方案，也不是为了做超重型分布式后端。  
它的目标很明确：

> 用尽可能清晰、可维护、可扩展的方式，解决 Unity 房间制联网项目的绝大多数工程问题。

## 架构概览

### 物理分层

- `Shared`：共享协议、基础设施、事件、绑定器
- `Server`：权威逻辑、房间、会话、模块、组件
- `Client`：客户端状态机、房间组件、回放、表现桥接
- `GameDemo`：示例玩法与演示场景

---

### 逻辑分层

#### Global
不依赖房间上下文的业务：

- 登录
- 大厅房间列表
- 建房 / 加房 / 离房
- 录像列表
- 下载录像

#### Room
依赖房间上下文的业务：

- 成员快照
- Ready 状态
- 游戏开始 / 结束
- 战斗同步
- 表情广播
- 局内聊天

---

### 标准数据流

```text
View 输入
  -> ClientApp.SendGlobal / SendRoom
  -> ServerApp 路由
  -> Server Module / RoomComponent 校验并修改权威状态
  -> S2C 广播 / 单播
  -> Client Module / ClientRoomComponent 收包
  -> LiteEventBus 抛出事件
  -> View 刷新 UI / 场景表现
```

## 典型开发流程

### 扩展一个房间玩法

以“房间发表情”功能为例，标准步骤如下：

1. 在 `Shared/Protocol` 中定义协议
2. 在 `Server/Components` 中编写服务端组件
3. 在 `Client/Components` 中编写客户端组件
4. 在 `StellarNetMirrorManager` 中注册组件工厂
5. 建房时把对应 `ComponentId` 加入组件清单
6. 在 View 层监听事件并做表现

这一套流程的核心收益是：

> 新玩法可以横向新增，不需要破坏已有房间逻辑结构。

## 为什么不用 SyncVar

因为多人项目真正难的不是“同步一个值”，而是：

- 同步从哪里来
- 谁改了状态
- 是否有越权
- 回放如何重建
- 重连如何恢复
- 排查问题时能否追到完整链路

StellarNet Lite 选择显式协议驱动，正是为了保证：

- 权威边界清晰
- 数据流向清晰
- 团队协作清晰
- 回放与重连逻辑可控

## 适合谁

这个项目更适合：

- 注重工程结构的 Unity 开发者
- 想做房间制联网游戏的独立团队
- 希望避免项目后期网络代码失控的主程
- 需要一套可扩展、可讲清楚、可维护的联机底座

不太适合：

- 只想用最少代码快速堆一个极简联机原型
- 完全依赖引擎黑盒同步工作流
- 超大型 MMO / 分布式后端场景

## 结语

StellarNet Lite 并不试图成为“最大的 Unity 网络框架”。  
它更关心的是另一件事：

> 当项目从 Demo 进入商业化迭代后，你的网络代码还能不能继续稳定扩展下去。

如果你也认同这些原则：

- 不信任客户端
- 不迷信黑盒自动同步
- 不接受房间逻辑写成巨石类
- 希望回放、重连、排障都讲得清楚

那么这套框架就是为这种项目准备的。

欢迎使用、扩展与持续完善。
